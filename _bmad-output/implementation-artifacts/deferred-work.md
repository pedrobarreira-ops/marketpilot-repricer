## Deferred from: code review of 1-1-scaffold-project-two-service-coolify-deploy-composed-health (2026-05-01)

- **pg Pools never `.end()`ed; no SIGTERM handler in app or worker** — Both `app/src/routes/health.js` and `worker/src/jobs/heartbeat.js` create module-scope `Pool` instances at import time and never close them. `worker/src/index.js` has no SIGTERM/SIGINT handler. Story 1.1 spec line 220 explicitly says the inline `pg` Pool is intentional scaffolding "not a pattern to replicate in later stories." Graceful shutdown + Pool teardown belongs to a later story (likely 2.1 when `shared/db/service-role-client.js` is formalized).
- **/health endpoint not behind `@fastify/rate-limit`** — Plugin is a runtime dep at AC#1 but spec does not register it at Story 1.1. A hostile burst on `/health` could exhaust the `max: 2` pool. Harden in a later security-pass story (Epic 11/12 ops or a dedicated edge-hardening story).
- **`@fastify/cookie` and `@fastify/csrf-protection` not registered** — Listed as runtime deps for AC#1, registration intentionally later. Owner: auth/session story (Story 1.5 founder-admins or onboarding stories that introduce sessions).
- **`MASTER_KEY_BASE64` shape (32-byte base64) not validated** — `shared/config/runtime-env.js` only checks presence, not decodability or 32-byte length. Story 1.2 explicitly owns this per Story 1.1 spec line 401: "Story 1.2 extends this module to add 32-byte length validation on `MASTER_KEY_BASE64`."
- **`.env.example` lists Stripe/Resend keys not in `REQUIRED_VARS`** — `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `RESEND_API_KEY` appear in `.env.example` but `getEnv()`'s required list is only `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_DATABASE_URL`, `MASTER_KEY_BASE64`. Intentional — they're needed at SDK init time only. Tighten in the story that first integrates Stripe (Story 11.1) or Resend (Story 12.2).
- **RLS comment in migration is unverified claim** — `db/migrations/202604301212_create_worker_heartbeats.sql` lines 7-10 claim Supabase's auto-RLS-trigger handles security on `worker_heartbeats`. Verify via Supabase project settings (Studio → Database → Tables → RLS toggle) rather than in code. Owner: any story that audits Supabase project settings (Epic 2 RLS work is a natural fit).
- **No `db:migrate` script in `package.json`** — Smoke test assumes the migration is already applied; nothing in scripts runs it. Pedro currently runs `npx supabase db push` manually per Task 3. Add a `"db:migrate": "npx supabase db push"` (or equivalent) when the team standardizes migration workflow — likely Story 4.1 when more migrations land.
- **Module load order — `health.js` constructs Pool before `getEnv()` validates env** — `app/src/server.js:6` imports `healthRoutes` which evaluates `health.js` and constructs the `Pool` *before* line 8's `getEnv()` runs. Low impact in practice (a Pool with `undefined` connectionString sits idle and `getEnv()` exits before `fastify.listen`), but worth tracking for the day env-validation moves earlier in the boot sequence (e.g., when `shared/db/service-role-client.js` becomes a factory in Story 2.1).

## Deferred from: code review of 1-2-envelope-encryption-module-master-key-loader-secret-scanning-hook (2026-05-01)

- **base64 trim asymmetry under non-documented openssl wrapping** — `shared/crypto/master-key-loader.js:45` round-trip uses `raw.trim()` which strips only leading/trailing whitespace; `re-encode` never has interior whitespace. Theoretical concern for very long base64 (>76 chars where openssl wraps); 32-byte keys produce 44 chars, so never triggered by the documented `openssl rand -base64 32`. Defense-in-depth fix would be to strip all whitespace before decoding.
- **secret-scanner pattern coverage is narrow (5 patterns)** — only catches MASTER_KEY-style assignment, shop_api_key assignment, sk_live_, sk_test_, Authorization Bearer. Real Stripe/GitHub leak shapes (restricted-scope keys with dots, scoped tokens) and Supabase service-role JWTs (`eyJ...`) are not covered. Spec-mandated 5 patterns at MVP; broaden in a later security pass (Epic 11/12 ops or pre-launch hardening). `.env.example` enumerates the gap (`SUPABASE_SERVICE_ROLE_KEY`, `STRIPE_WEBHOOK_SECRET` whsec_, `RESEND_API_KEY` re_, `SUPABASE_SERVICE_ROLE_DATABASE_URL` postgres://).
- **single opaque `KeyVaultDecryptError` masks programmer-error vs tamper** — `shared/crypto/envelope.js:82-90` validation branch and `:99-102` crypto-failure branch both throw `KEY_VAULT_DECRYPT_FAILED`. Spec-mandated for attacker-oracle protection but conflates programmer error with security event in metrics/alerts. If alerting on decrypt failures becomes important, split classes.
- **master key buffer not zeroed on shutdown; no `inspect`/`toString` guard** — `worker/src/index.js:15` declares `let masterKey;` at module scope; no `process.on('exit', () => masterKey.fill(0))`. Buffer persists in heap dumps. Also no override of `inspect`/`toString` to prevent accidental debug-log dump. Defense-in-depth beyond MVP.
- **returned EncryptedKey buffers are by-reference** — `shared/crypto/envelope.js:60-65` returns the live `nonce`/`ciphertext`/`authTag` Buffers. Caller mutation between encrypt and persist silently corrupts vault state. Low priority; freeze or copy in a hardening pass.
- **`loadMasterKey` trailing-whitespace edge cases** — `shared/crypto/master-key-loader.js:32,45` mixes `raw` and `raw.trim()`; correct today but fragile. Tighten in a future refactor.
- **`updated_at` has DEFAULT but no auto-update trigger** — `supabase/migrations/202604301204_create_shop_api_key_vault.sql:31` requires every writer to remember `SET updated_at = NOW()`. Add a `BEFORE UPDATE` trigger when the second writer ships (Story 4.3).
- **multi-line secret values bypass line-by-line scanner** — JS template literals or bash heredocs that split a secret across lines defeat all 5 patterns. Inherent to diff-line scanning. Document the limitation in `scripts/check-no-secrets.sh` header.
- **RLS policy silent-fail on `auth.uid()` NULL** — `supabase/migrations/202604301204_create_shop_api_key_vault.sql:42-44` returns 0 rows for anon (correct, but by SQL NULL semantics, not explicit). Add `auth.uid() IS NOT NULL AND ...` defensive clause when Story 2.2's RLS regression suite lands. (Forward-pointer comment already in the migration.)
- **`last_failure_status smallint` accepts negatives and >599** — `supabase/migrations/202604301204_create_shop_api_key_vault.sql:29` no CHECK constraint. Add `CHECK (last_failure_status BETWEEN 100 AND 599)` if the column is meant to be HTTP-only. No analytics depend on it yet.
- **`masterKey` variable is module-scope dead weight at boot** — `worker/src/index.js:15-25` loads and logs but never passes the buffer to any consumer. Story 7.x must thread it through `startHeartbeat` (or successor jobs) — current shape is an architectural placeholder, not a runtime bug.
- **over-padded base64 round-trip accepted; saved only by the 32-byte length gate** — `shared/crypto/master-key-loader.js:44-47` strips trailing `=+` from both sides before comparing, so over-padded inputs slip past the round-trip but fail the `decoded.length !== 32` check. Acceptable today; would silently accept malformed input if AES-128 ever became an option.
- **`withEnv` test helper not robust under concurrent runners** — `tests/shared/crypto/master-key-loader.test.js:7-17` uses `process.env` (process-wide) — fine while `node:test` runs sequentially, breaks if `--concurrency=N` is ever set. Use a stub-env-via-Reflect-defineProperty pattern when test parallelism matters.
- **`.gitattributes` doesn't auto-renormalize pre-existing CRLF working trees** — `.gitattributes:4-5` enforces LF for fresh clones, but Pedro's existing checkout (made before this commit) may still have CRLF in `.githooks/pre-commit`. If Git Bash chokes with `bad interpreter: /usr/bin/env\r`, run `git add --renormalize . && git commit -m "renormalize line endings"`. Document in README if it bites.
- **hook bypassed via GitHub web UI / Desktop GUI / `git commit --no-verify`** — `.githooks/pre-commit:4` is local-only. Pedro must enable GitHub repo Settings → Code security → Secret scanning as the second layer (mentioned in README). The hook is the FIRST line of defense, not the only one.
- **coverage gap on extreme plaintext sizes in envelope tests** — `tests/shared/crypto/envelope.test.js` covers happy-path-size strings only; no `plaintext.length === 1` or `>1 MB`. Code is correct (GCM `update` invariant); coverage gap only. Tighten in a future test-pass story.
- **hook does not fire on `git rebase --continue` / `cherry-pick` / `merge`** — `.githooks/pre-commit` only runs on `git commit` and `git commit --amend`. Secrets introduced via cherry-pick from a feature branch escape local scanning entirely. Document in `scripts/rotate-master-key.md` (or a developer ops doc) that rebase/cherry-pick paths require a manual `bash scripts/check-no-secrets.sh` before push.
- **pathspec `:!*.md` exclusion is git-version-dependent** — `scripts/check-no-secrets.sh:42` works on git ≥2.13 (current Git for Windows is fine); old CI runners with git <2.13 would treat `*.md` as top-level only. Pin git version in CI or use `:(exclude,glob)*.md` explicit form when CI is set up.
- **back-port the two regex transcription-bug fixes to source distillate** — `_bmad-output/planning-artifacts/epics-distillate/01-epics-1-3-foundation-tenancy-mirakl.md` AC#4 still has the broken `MASTER_KEY[A-Z_]*` (no digits) and `Authorization:[ \t]*Bearer` (no optional quote) regex text. Future re-shardings will inherit the bugs. Update both lines to match the corrected forms in `scripts/check-no-secrets.sh`. Housekeeping; outside Story 1.2's commit scope.
- **worker has no SIGTERM handler; masterKey not zeroed on shutdown** — also covered by Story 1.1's deferred "pg Pools never `.end()`ed; no SIGTERM handler" item. Bundle into the same shutdown-discipline story (likely 2.1 or a dedicated ops story).
