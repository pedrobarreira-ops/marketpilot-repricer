## What Hurt — Reframed

Bob's first read framed five themes; Pedro's reframe surfaced two more from outside the story Dev Notes.

### 1. First confirmed slipped Epic-1-retro item: distillate directory-tree drift

`architecture-distillate/05-directory-tree.md` line 183 still says `db/migrations/` — the actual code directory is `supabase/migrations/`. Story 2.2's Dev Notes flagged the drift mid-implementation (forced Amelia to override a distillate truth in code).

This is the first **confirmed slipped** item from Epic 1's retro action plan. Other Item 1 sub-items shipped: regex back-port ✅ in Story 1.2 AC#4, Library Empirical Contracts hoist ✅ in `project-context.md` (now ~12 entries), Story 1.4 AC#3 amendment ✅ citing "Q4 of Epic 1 retro 2026-05-03." The directory-tree correction simply didn't make it into the housekeeping PR.

**Reframe**: small, mechanical to close, and the slip itself doesn't blast Epic 3 — but it's the first concrete instance of an Epic-N retro action item leaking past Epic-N+1 prep. Worth registering as a baseline rate, not as failure. Action: fix in Item 1 below before Epic 3 sharding starts.

### 2. Stories 1.4 + 1.5 Cloud-migration push not confirmed

No Supabase MCP verification record visible. Local-only verification through Epic 1 + Epic 2 was acceptable; Epic 3 raises the bar because Story 3.3 (`mirakl-empirical-verify`) chains through Cloud Supabase and Story 3.x integration tests will need the schema present. **Reframe**: this isn't tech debt — it's an unfinished Pedro-side ops task carried over from Epic 1 retro's readiness assessment. Now P0 for Epic 3 prep (Item 2 below).

### 3. BAD Step 6 PR-body hallucinations are a pattern, not noise

Two PRs in a row hallucinated PR-body facts:

- **PR #64**: file count off-by-one + invented `DATABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` env-var pair that wasn't in the diff
- **PR #65**: false `RUN_CI_LOCALLY=true` skip claim while GitHub Actions actually ran and was green

Cosmetic each time, but it makes PR bodies untrustworthy as documentation. Two-for-two is a pattern.

**Reframe**: this is upstream BAD-tooling friction, NOT story-execution friction. Story Dev Notes don't capture it because the PR body is generated outside the story's lifecycle. The fix is data-grounding Step 6 (give it `git diff --stat` output and `gh pr checks` output to copy from) — removing the hallucination opportunity rather than asking the model harder. Action: Item 4 below.

### 4. Sprint-status.yaml conflicts at PR-merge time

PR #65 required force-push + rebase to clear a `sprint-status.yaml` conflict between PR branch and `main`. Root cause: BAD subagents commit sprint-status flips on the feature branch AND on `main` directly during a story's lifecycle, so they diverge. Recurs on every PR unless the dual-flip pattern changes.

**Reframe**: another upstream BAD-tooling pattern, not story execution. Pedro's "who reads sprint-status mid-story?" test is the right cut — the "live state on the feature branch" benefit is theoretical (BAD's coordinator reads it at Phase 0 + Phase 4 batch boundaries; Amelia, Bob, Quinn don't). Cost compounds: ~60 future stories × 1 force-push/rebase per PR = 60 chances to mess up a judgment-laden conflict resolution. Trade 60 future conflicts for one upfront subagent-prompt patch. Decision recorded as Q2 below; implementation in Item 5.

### 5. BAD Phase 4 worktree-cleanup leaked silently on PR #65

PR #64's post-merge report explicitly said "Worktree and branch cleaned up." PR #65's post-merge report didn't mention worktree cleanup at all. Pedro cleaned it up manually mid-retro (`git worktree remove` + `git branch -D`).

**Substantive issue**: bad-review's Phase 4 safe-merge cleanup either silently skipped on PR #65 or was bypassed by the manual merge path. Real BAD-pipeline friction. **Reframe**: investigate the divergence in `bad-review/SKILL.md` Phase 4 — figure out whether cleanup is unconditional or path-dependent, patch if needed. Action: Item 3 below.

### 6. Story-file filename-convention drift between 2.1 and 2.2

- Story 2.1 file: `2-1-rls-aware-app-db-client-service-role-worker-db-client-transaction-helper.md`
- Story 2.2 file: `story-2-2-rls-regression-suite-ci-block.md` (extra `story-` prefix)

Same Bob, same epic, two different filename conventions. CR/BAD scripts that grep by `<epic>-<story>-*.md` would miss the 2.2 file.

**Reframe**: per Pedro's "challenge premature abstractions before designing solutions" rule — one repeat is a watch item, not a fix item. Trigger to revisit: Story 3.x lands with a third filename convention OR a CR/BAD script breaks on the inconsistency. Watch List, not Action Items.

### 7. Story Dev Notes don't capture upstream-tooling friction (meta-observation)

Three of the friction items above (#3, #4, #5) aren't in the story records by construction — they're failures in BAD pipeline orchestration / bad-review SKILL.md, which lives outside the story lifecycle. Story Dev Notes capture what happens *during* implementation, not what fails *around* implementation.

**Reframe**: real gap in the retro mechanism. Worth flagging for next retro: when synthesizing patterns, supplement story-records analysis with a pass over recent PR threads and BAD/bad-review skill files. Watch item, no action this cycle.

### 8. `shop_api_key_vault` test sequencing creates Story-4.1 forward-debt

Story 2.2 had to defer the row-level isolation test for `shop_api_key_vault` to Epic 4 because `customer_marketplaces` (the FK target) doesn't exist yet. The deferred test (`rls_isolation_shop_api_key_vault_deferred_awaiting_epic4`) is a placeholder that asserts the policy *exists* but not that it *works at the row level*. The convention test reads the seed — `shop_api_key_vault` isn't in the seed yet — so the convention safety net is also dormant for that table.

**Reframe**: forward-debt that must be paid when Epic 4 ships `customer_marketplaces`. Three things in three different files (seed extension, registry extension, deferred-test replacement). Risk: if any one is forgotten, the table ships RLS-untested. Mitigation: explicit Story-4.1 forward-checklist note in `sprint-status.yaml` now while it's fresh. Action: Item 6 below.

### 9. Multi-table migration RLS-policy detection has a known weak spot

Review patch on Story 2.2 caught it: `negative_assertion_no_migration_missing_rls_policy` does a global per-file check, not a per-table check. A migration that creates two tables and adds RLS policy on only one would silently pass for both. The patched version tightens via `.deferred` glob filtering, but the underlying limitation remains.

**Reframe**: honest tech debt. Not bug-tier, not blocking, but a place where the test pattern's assumption ("one migration = one table") will break when Epic 4+ ship multi-table migrations. Watch item with trigger: first multi-table migration in Epic 4+.
