## How to use these references

**From Architecture documents.** Cite by filename when describing surface-level concerns:

> The repricer-worker's circuit-breaker trip emits a customer-facing event surfaced via the dashboard banner system (see `_bmad-output/design-references/screens/06-audit-log-root.html` Atenção feed and `screens/03-dashboard-live.html` banner zone).

**From Epics / Stories.** Cite by filename + skeleton-section to anchor implementation:

> **Story 5.3 — Audit log search-by-SKU**
> Implement Surface 4 of the audit log per `screens/12-audit-search-active.html`. Reference: skeleton §4.1.4 (UX10), FR38.

**From BAD subagents.** When a story references a screen file, the dev subagent reads the HTML to understand the visual target. Until real HTML lands, the placeholder + skeleton-section reference is sufficient for non-visual implementation work (engine, integration, schema, RLS). Pure UI implementation should wait until real HTML is committed.

## How to update once Claude Design resets

1. Generate Phase D's three artboards (`13-admin-status`, `14-key-revoked`, `15-payment-failed`) per the Phase D prompt drafted by Sally in conversation history.
2. Ask Claude Design to package the full canvas (now 16 artboards) as standalone HTML.
3. Replace placeholders. Two viable options:

   **Option A (preferred)** — Extract each artboard's section into its own file, replacing the placeholder one-by-one. References stay valid. ~30 min one-time work.

   **Option B (faster)** — Commit the single packaged HTML as `claude-design-mockups.html` and update each `screens/*.html` to redirect to the corresponding anchor (`<meta http-equiv="refresh" content="0; url=../claude-design-mockups.html#artboard-02">`). References stay valid via redirect.

Option A keeps each screen viewable in isolation. Option B is faster and preserves Claude Design's interactive tweaks toolbar.

## What this folder does NOT contain

- **Implementation code** — lives in `app/`, `worker/`, etc. (per `_bmad-output/planning-artifacts/prd.md` topology).
- **Design intent / decision rationale** — lives in `free-report-design-chat.md` (this folder) and the UX skeleton.
- **Authoritative behavior contract** — lives in `../planning-artifacts/prd.md`.
- **PT microcopy verbatim** — lives in `../planning-artifacts/ux-skeleton.md` §9.

If you need to know **what** a screen does, read the PRD. If you need to know **why** a design choice was made, read the UX skeleton or the design-intent transcript. If you need to know **how it should look**, open the matching `screens/*.html` (placeholder for now, real HTML after Claude Design pass completes).
